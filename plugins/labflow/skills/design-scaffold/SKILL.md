# Design Scaffold

本 skill 主要用于**设计阶段**，服务于实现阶段的**分布式 prompt**：把用户脑中的想法外化到代码骨架、字段文档、TODO 锚点或设计 note 中，让后续实现不必依赖一段集中式长 prompt。

它不是“写一段完整任务 prompt”，也不局限于声明式配置。声明式 dataclass / Cfg 是常见风格，但核心能力是：和用户共同把概念、边界、约束、字段语义、数学关系和实现锚点分散落到最贴近实现的位置。

## 适用场景

- 用户有一个 idea，但还没完全成型。
- 需要设计模块职责、接口草图、配置字段或 runtime 壳。
- 需要把口头需求写成代码内可维护的说明。
- 需要字段级长文档、数学约束、示例、边界条件。
- 需要给后续实现留下清晰 TODO 伪算法。

不适用：

- 已经明确要求直接实现 bug fix 或 feature。
- 纯注释润色；使用 `annotation`。
- 纯算法伪代码骨架；使用 `pseudocode`。

## 工作方式

### 1. 先共识，不急着实现

先把已知信息压成短共识：

- 目标：这个设计解决什么问题。
- 边界：它不负责什么。
- 输入/输出：用户、配置、runtime 分别看到什么。
- 约束：数学、物理、数据形状、兼容性、默认值。
- 开放问题：哪些点仍需用户判断。

可从代码库中发现的事实先自行调研；不要把“在哪里定义/怎么注册”这类可发现问题丢给用户。

### 2. 落成代码脚手架

根据上下文选择合适形态：

- dataclass / Cfg 字段。
- TypedDict / Pydantic / schema。
- runtime class 壳。
- 函数签名与 docstring。
- 模块级说明。
- TODO 伪算法块。
- markdown 设计 note。

默认不写完整实现；如果用户明确要实现，再进入 coding。

### 3. 字段说明要承担设计沟通

字段文档不只是解释类型，而要回答：

- 这个字段控制哪一层概念。
- 它和相邻字段的区别是什么。
- 默认值为什么合理。
- 支持哪些简写和详细形式。
- 边界条件、非法组合、数值含义是什么。
- 若涉及数学对象，用 LaTeX 明确写出。

### 4. 保留未来实现锚点

在脚手架中留下后续 coding 可直接执行的锚点：

```python
# TODO: 实现采样分布分派
# 输入：...
# 输出：...
# 约束：...
# 验收：...
```

这些 TODO 是设计规格，不是临时废话。后续实现必须迁移或保留其信息量。

## 输出标准

好的 design scaffold 应该：

- 像代码库的一部分，而不是像聊天记录。
- 用户能逐段审阅并指出“这里不对”。
- 后续 coding agent 能据此实现，不需要重新猜需求。
- 不把所有需求集中压进对话窗口，而是分布到相关代码/文档位置。
- 保留用户审美和控制力，agent 负责补全表达与结构。

## 常见模式：声明式配置

当项目偏声明式配置驱动时，优先考虑：

- `@dataclass` Cfg 表达用户可调意图。
- 字段下方用 `r"""..."""` 长文档说明概念层级、数学关系和示例。
- `__post_init__` 或 validator 表达局部校验。
- runtime class 只消费已经规范化的配置。

这只是默认偏好，不是强制框架。
